# Overview

This guide describes how to prepare **RIEGL TLS** data for generating vegetation profiles using **pylidar**. It assumes that the scans have already been co-registered using recommended workflows:

- **VZ‑400i:** https://github.com/qforestlab/riscan_registration/tree/main
- **VZ‑400:** https://github.com/qforestlab/riscan-registration-VZ400

---

## Required Input Files

For each scan position, ensure the following files are available:

| File Type | Description | Source |
|---------|-------------|--------|
| **.rxp** | Raw TLS scan data. | Created on scanner / imported into `.RiSCAN` project. |
| **.rdbx** | Processed scan data including MTA correction. | Generated by converting `.rxp` in RiSCAN PRO. |
| **.dat** | Scanner position & orientation. | Exported after co‑registration. |

---

## 1) RXP Files

The `.rxp` file contains raw LiDAR returns, intensities, timestamps, and acquisition metadata.

These files are automatically generated by the scanner and retained when imported to the `.RiSCAN` project.

---

## 2) RDBX Files and MTA Processing

The `.rdbx` file contains scan data processed to correct for **Multiple Time Around (MTA)** ambiguities. These occur when pulse emission rates are high and multiple laser pulses overlap in time, common in forest environments and high-frequency scanning.

### Disable GPU for Conversion

Reproducibility requires converting `.rxp` → `.rdbx` using **CPU-only** processing.

1. Open your project in **RiSCAN PRO**
2. Disable GPU processing:  
   `Tools → Settings → General → Data Acquisition`
   
![](./img/gpu_settings.png)

3. Re-convert the scans:  
   Select scans → Right-click → **Convert**

![](./img/select_all.png)
![](./img/convert.png)

---

## 3) DAT Files (Sensor Orientation & Position)

The `.dat` files store the **scanner’s position and orientation**, required for vegetation profile calculation.

These files are exported after co‑registration. Ensure they are stored in the `.RiSCAN` project folder.

![](./img/SOP_export.png)

## Why This Matters

Gap-fraction methods rely on knowing both where each laser pulse originated and how many returns it produced as it passed through the canopy. By linking these two pieces of information, we can reconstruct the path of the pulse and determine where vegetation intercepted it. This makes the accuracy of the return information critical.

Modern TLS systems such as the VZ-400i can operate at very high pulse rates (up to ~1200 kHz). At these frequencies, correctly resolving the sequence of pulse returns requires Multiple Time Around (MTA) processing, which is applied when converting .rxp files to .rdbx.

RiSCAN PRO can perform MTA processing using either the CPU or GPU. However, we have observed that GPU-based processing can produce inconsistent return assignments between conversions. To ensure reproducibility and stable return classification, we therefore always disable GPU processing and perform the .rxp → .rdbx conversion using CPU only.

### GPU vs CPU Reproducibility

RiSCAN PRO's GPU-based MTA processing may produce **inconsistent return handling**. CPU conversion produces stable and reproducible results.

To verify, inspect the **convert logs**, located in:

```
project.RiSCAN/project.rdb/SCANS/ScanPos001/SINGLESCANS/<scan_timestamp>/
```

Scroll to the **Statistics** section:
- The number of **echoes per processing zone** should remain identical across repeated conversions

If these values vary, GPU processing was used and must be disabled.

![](./img/convert_logs.png)
